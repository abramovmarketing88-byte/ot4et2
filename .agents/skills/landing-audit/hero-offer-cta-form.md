# Шаблон: первый экран, оффер, CTA и форма

Используй как чек-лист или промпт. Прогон **обязателен** в каждом Pass 1 (и повторно после правок hero/оффера/CTA/формы).

Если на вопрос нельзя ответить по странице — статус ❓, задай вопрос пользователю. Не додумывай факты бизнеса.

***

## 2.1. Анализ первого экрана

Ответь на 5 вопросов:

1. Что здесь продают (одним предложением, без красивых слов)?
2. Для кого это (тип клиента, сегмент, объёмы, ситуация)?
3. В чём главная польза/результат для клиента?
4. Чем этот оффер лучше/выгоднее, чем у типичного конкурента?
5. Какое действие хотят от пользователя прямо сейчас?

Если хотя бы на один из этих вопросов ты не можешь ответить за 5 секунд — первый экран слабый, надо переписывать заголовок/подзаголовок и упростить.

Структура сильного первого экрана:
- Заголовок: “Что + для кого” (без воды).
- Подзаголовок: “Как именно делаем и чем выгодно” (цифры, форматы, сроки).
- 1–2 буллета с УТП: объёмы/сроки/гарантия/формат оплаты.
- CTA + пояснение: “Получить расчёт / забронировать дату / запросить выезд”.
- Визуал, показывающий контекст услуги.

### Действия агента

| Результат 5 вопросов | Действие |
|----------------------|----------|
| Все 5 ясны со страницы | Оцени качество формулировок; правь только если слабые |
| 1+ вопрос без ответа | Переписать hero (заголовок/подзаголовок/буллеты/CTA) |
| Нет фактов (цифры, сроки, УТП) | ❓ пользователю → после ответа внести в код |
| Нет фото контекста | ❓ или заменить на лучший имеющийся ассет / плейсхолдер |

## 2.2. Анализ оффера

Прогони через мини-шаблон:

1. Что конкретно получает клиент (конечный результат/формат)?
2. В какие сроки он это получает (реалистичный диапазон, а не “моментально”)?
3. При каких условиях (объём, минимальный чек, регион, тип клиента)?
4. Чем это отличается от стандартного игрока рынка (как минимум 1–2 пункта)?
5. Есть ли ограничение/дефицит (даты, количество объектов, сезон)?
6. Понятна ли цена или хотя бы порядок (от/диапазон/расчёт по фото)?

Если оффер слишком размытый, его нужно “сжать” до одной чёткой формулы и на неё уже навешивать детали.

### Формула сжатия (после ответов)

```
[Результат] для [сегмент] за [срок] — [отличие] — [условие/от] — [следующий шаг]
```

Пример паттерна: результат + срок + снижение риска (если уместно и подтверждено клиентом).

## 2.3. Анализ CTA

Простой чек-лист для кнопок:

1. Можно ли, закрыв текст, по надписи на кнопке понять, что произойдёт?
2. Глагол в CTA активный и конкретный (“Получить расчёт”, “Записаться”, “Заказать выезд”).
3. CTA не дублирует обезличенное “Отправить” или “Ок”.
4. Кнопка визуально заметна (цвет, размер, расположение).
5. Нет конкурирующих кнопок рядом одинаковой важности.
6. На мобильном CTA в зоне большого пальца (особенно главный).

Если ответ “нет” хотя бы на 2–3 пункта — переписать CTA и/или поменять визуальный приоритет.

### Правила правок CTA

- Один смысл главного CTA на всей странице.
- Под кнопкой — микрокопирайт «что дальше» (срок ответа / формат).
- Второстепенные («смотреть кейс», «в мессенджер») — visually secondary.
- Не менять endpoint форм/телефонов без проверки.

***

## 2.4. Анализ формы

Шаблон вопросов:

1. Все ли поля действительно нужны на первом шаге?
2. Объяснено ли клиенту, зачем каждый блок данных (телефон, e-mail, фото и т.д.)?
3. Легко ли ввести данные с телефона (маски, тип клавиатуры, длина формы)?
4. Понятно ли, что будет после отправки и в какой срок с ним свяжутся?
5. Есть ли альтернатива для тех, кто не любит формы (мессенджер, звонок, “позвоните мне”)?
6. Технически всё работает, заявки не теряются, уведомления приходят?

Часто +20–30% к конверсии даёт один шаг: урезать форму до 2–3 полей и добавить нормальное объяснение “что дальше”.

### Действия агента

| Проблема | Фикс |
|----------|------|
| Слишком много полей | Урезать до 2–3; лишнее — на второй шаг / менеджеру |
| Нет подсказок | Добавить hint / placeholder по смыслу |
| Нет «что дальше» | Текст под формой + success-state |
| Нет альтернатив связи | Добавить tel / messenger, если есть контакты (❓ если нет) |
| Неясно, доходит ли заявка | ❓ или тестовая отправка; не ломать CRM-хуки |
